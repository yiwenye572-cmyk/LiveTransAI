## 一、项目概述

### 1.1 目标

开发一款 **AI 同声传译助手**，帮助用户实时观看英文（或其他外语）演讲、技术分享、国际会议或网课时，将单向音频流实时翻译成中文，以字幕形式呈现。核心差异化能力：**自动纠正之前识别或翻译的错误**。

### 1.2 核心价值主张

> 现有同传产品通常以**低延迟前向流式字幕**为核心体验，用户看到的是当前时刻模型给出的最佳结果。我们的差异化方向是在不牺牲实时性的前提下，增加一条后台校对通道：当更多上下文到达后，系统可以对已经展示过的字幕做**可感知、可追踪的回溯修正**。

这个定位更适合实训项目：不宣称替代底层同传模型，而是在系统编排层面补足"实时字幕缺少上下文复核"的问题。

## 二、行业调研与差异化分析

### 2.1 现有产品对标

| 产品 | 类型 | 翻译质量 | 展示后修正能力 | 中文优化 |
|------|------|----------|----------------|----------|
| 豆包 App 同传 | 消费级 | ⭐⭐⭐⭐ | 未观察到显式回溯修正 | ✅ |
| OpenAI Realtime Translation | API | ⭐⭐⭐⭐ | API 层通常以前向增量为主 | ⭐⭐⭐ |
| Wordly | 企业 SaaS | ⭐⭐⭐⭐ | 主要依赖术语表/人工配置 | ⭐⭐⭐ |
| Interprefy | 企业级 | ⭐⭐⭐⭐ | 主要面向会议同传流程 | ⭐⭐⭐ |
| Google Meet 翻译字幕 | 免费 | ⭐⭐⭐ | 字幕体验以前向显示为主 | ⭐⭐⭐ |
| Deepgram + LLM | API 拼接 | ⭐⭐⭐ | 需要应用层自行实现修正 | ⭐⭐ |
| **本项目** | 实训作品 | ⭐⭐⭐⭐ | ✅ 应用层 revision + Diff 修正 | ✅ |

### 2.2 豆包同传 API 协议分析

通过逆向分析豆包 AST（同声传译）API 的 Protobuf 协议，确认：

- 事件类型仅有：`SourceSubtitleResponse`（原文）、`TranslationSubtitleResponse`（译文）、`TTSSentenceStart/Response/End`（语音）
- **没有任何 `Correction`、`Revision`、`Update` 类型的事件**
- 客户端只能被动接收前向流式输出，无法向服务端请求"请重新翻译第 N 句"

> 结论：豆包 AST 当前协议没有提供"修改历史字幕"的原生事件。我们的机会不是改变 AST，而是在应用层引入 `revision` 模型，对已展示字幕做可解释的二次校对与替换。

### 3.2 设计决策：为什么是本地后端而不是服务端部署

| 考量 | 本地后端 | 服务端部署 |
|------|----------|------------|
| 端到端延迟 | 0ms（localhost） | +200~300ms 网络延迟 |
| 系统音频捕获 | ✅ sounddevice WASAPI 回采 | ❌ 服务器无法抓用户声卡 |
| 用户隐私 | ✅ 音频不经过自建服务端和前端；仅直连第三方 AST | ❌ 自建服务端需要中转用户音频 |
| 部署复杂度 | `python main.py` | 需 Docker/K8s/域名 |
| 豆包 AST 连接 | 用户电脑直连 | 服务器中转增加延迟 |

> 核心原则：**音频采集、前端渲染和会话控制放在本地；需要模型能力的识别、翻译、纠错走云端 API。** 这样可以减少自建服务端的隐私和部署压力，同时保留第三方模型能力。

## 四、核心模块设计

### 4.1 音频采集模块（audio/capture.py）

**职责**：捕获系统音频输出（声卡回采）或麦克风输入，产出 80ms PCM 音频帧。

**设计原则**：默认使用系统音频回采。这是题目的核心场景——用户观看英文演讲/Zoom/网课时，音频从电脑扬声器播放，系统自动抓取，用户**零操作**。

#### 4.1.1 工作原理

```
YouTube/Zoom 播放音频 → Windows 声卡 → 扬声器（用户听到声音）
                                       │
　　　　　　　　　　　　　　WASAPI Loopback 设备 ← sounddevice 抓取
                                       │
                              16kHz 单声道 PCM 音频帧
                                       │
                              豆包 AST WebSocket
```

Windows 上通过 **WASAPI Loopback** 设备抓声卡输出。macOS 上通过 **BlackHole** 虚拟声卡路由系统音频。

#### 4.1.2 技术选型对比

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| PyAudio + WASAPI | 简单，pip 安装 | 部分 PyAudio 版本不支持 loopback | ⚠️ 需要验证 |
| **sounddevice** | 原生支持 WASAPI loopback，API 更现代 | 需要额外安装 | ✅ **推荐** |
| PortAudio (C 绑定) | 最底层控制 | 开发量大 | ❌ 3 天工期不现实 |

**推荐使用 `sounddevice` 库**，它基于 PortAudio，原生支持 WASAPI Loopback 且 API 简洁。

#### 4.1.3 四种场景覆盖

| 场景 | 音频来源 | 采集方式 | 用户操作 |
|------|----------|----------|----------|
| A: 看 YouTube 英文演讲（**主场景**） | 浏览器 → 声卡 | WASAPI Loopback | ✅ 零操作 |
| B: 参加 Zoom 英文会议 | Zoom → 声卡 | WASAPI Loopback | ✅ 零操作 |
| C: 看本地英文视频/网课 | 播放器 → 声卡 | WASAPI Loopback | ✅ 零操作 |
| D: 用户自己说英语练习 | 麦克风 | 默认输入设备 | ⚠️ 切换模式 |

> 场景 A/B/C 对标题目"观看英语演讲、技术分享、国际会议或网课"——全部由系统音频回采覆盖。

#### 4.1.4 关键实现

```python
import sounddevice as sd
import numpy as np

class AudioCapture:
    """音频采集，默认系统音频回采"""
    
    def __init__(self, state: SessionState, mode: str = "loopback"):
        self.state = state
        self.mode = mode          # "loopback" | "microphone"
        self.rate = 16000         # 豆包 AST 要求
        self.chunk = 1280         # 80ms at 16kHz
        self.stream: sd.InputStream | None = None
    
    def _find_loopback_device(self) -> int | None:
        """查找 WASAPI Loopback 设备"""
        devices = sd.query_devices()
        for i, dev in enumerate(devices):
            if dev["max_input_channels"] > 0:
                # Windows: 名称含 "Loopback"
                # macOS: 名称含 "BlackHole"
                name = dev["name"].lower()
                if "loopback" in name or "blackhole" in name:
                    return i
        return None  # 未找到，降级为麦克风
    
    def start(self):
        device = None
        if self.mode == "loopback":
            device = self._find_loopback_device()
            if device is None:
                print("[Audio] Loopback 设备未找到，降级为麦克风")
                self.mode = "microphone"
        
        self.stream = sd.InputStream(
            device=device,
            samplerate=self.rate,
            channels=1,
            dtype='int16',
            blocksize=self.chunk,
            callback=self._on_audio_chunk
        )
        self.stream.start()
    
    def _on_audio_chunk(self, indata: np.ndarray, frames, time, status):
        """每 80ms 触发一次，将音频帧送入处理管线"""
        if self.state.phase == SessionPhase.RUNNING:
            # 转为 bytes，交由下游处理
            pcm_bytes = indata.tobytes()
            asyncio.run_coroutine_threadsafe(
                self._pipeline.feed(pcm_bytes), 
                self._loop
            )
    
    def pause(self):
        """暂停采集（SessionManager 调用）"""
        if self.stream:
            self.stream.stop()
    
    def resume(self):
        """恢复采集"""
        if self.stream:
            self.stream.start()
    
    def stop(self):
        """停止采集"""
        if self.stream:
            self.stream.close()
```

#### 4.1.5 已知坑 & 对策

| 坑 | 现象 | 对策 |
|----|------|------|
| 部分电脑无 Loopback 设备 | `device` 返回 None | 自动降级为麦克风模式，前端提示 |
| WASAPI Loopback 只有立体声 | 声道数 = 2 | 代码中取左声道或混音为单声道 |
| macOS 需装 BlackHole | 无系统级 Loopback | README 写清楚安装步骤；或降级麦克风 |
| 蓝牙耳机延迟 | Loopback 延迟大 | 建议用户用有线/内置扬声器 |

### 4.2 VAD 静音检测模块（audio/vad.py）

**职责**：检测当前是否有人在说话。**不做断句决策**——断句由豆包 AST 内部完成。

**提供给 FlowController 的能力**：
- `current_silence_duration()` → 当前持续静音了多久（秒）
- `is_speaking()` → 当前是否在说话
- VAD 结果比 AST 的 `AudioMuted` 事件响应更快

### 4.3 豆包 AST 客户端（translator/ast_client.py）

**职责**：建立 WebSocket 连接到豆包同声传译服务，发送音频流，接收翻译结果。

**接口地址**：`wss://openspeech.bytedance.com/api/v4/ast/v2/translate`

**鉴权方式**（新版控制台）：
```python
headers = {
    "X-Api-Key": "your-api-key",
    "X-Api-Resource-Id": "volc.service_type.10053"
}
```

**通信协议**：WebSocket + Protobuf 二进制序列化

**交互流程**：

**API 关键特性**：
- 支持 S2T（语音→文本）和 S2S（语音→语音）两种模式
- 支持 20 种外语 + 2 种方言（粤语、上海话）
- 输出包含 `spk_chg` 字段用于自动检测说话人切换
- 每个句子有 Start/Response/End 三个事件，Response 可能多次推送（句内增量更新）

**句内增量示例**：
```

TranslationSubtitleResponse #2: "大家好，今天"
TranslationSubtitleResponse #3: "大家好，今天我想谈谈AI"
TranslationSubtitleEnd:        "大家好，今天我想谈谈AI"  ← 最终版本
```

### 4.4 FlowController 流控决策层（controller/flow_controller.py）

**核心定位**：不重新断句，不替代 AST。而是在 AST 输出之上加一层**决策智能**。

**不做的事情**（AST 做得更好）：
- ❌ 声学层面断句
- ❌ 语言层面端点检测

**做的事情**（AST 不知道也不做的）：
- ✅ 碎片合并：AST 误切时合并展示
- ✅ 纠错触发决策：什么时候发起后台精翻
- ✅ UI 状态管理：比 AST 更快通知前端"等待中"

**决策规则**：

```python
class FlowController:
    def __init__(self):
        self.pending = []          # AST 输出但尚未确认展示的句子片段
        self.displayed = []        # 已展示且可被纠错的稳定句子
        self.last_correction_time = 0
        self.commit_timer = None
        self.min_wait_ms = 300     # 给短碎片留一点 lookahead 时间
        self.max_wait_ms = 800     # 最大等待时间，保证实时性

    def on_ast_sentence_end(self, source_text, translation, start_time, end_time):
        """
        AST 宣布一句结束后，不立刻清空 pending。
        先判断是否需要等待下一片段，避免短句误切直接展示。
        """
        self.pending.append({
            "source": source_text,
            "translation": translation,
            "start": start_time,
            "end": end_time,
            "received_at": time.time()
        })

        # ── 决策 1: 展示、等待或合并？ ──
        if self._should_commit_now():
            self._commit_pending()
        else:
            self._schedule_delayed_commit()

        # ── 决策 2: 是否触发后台纠错？ ──
        if self._should_trigger_correction():
            window = self.displayed[-8:]
            asyncio.create_task(self.correction_engine.run(window))

        # ── 决策 3: UI 状态？ ──
        self._update_ui_status()

    def _should_commit_now(self):
        """
        合并策略：短碎片可以短暂等待，但不能无限等待。
        核心目标是平衡字幕实时性与阅读完整性。
        """
        if not self.pending:
            return False

        first = self.pending[0]
        waited_ms = (time.time() - first["received_at"]) * 1000

        # 规则 1: 超过最大等待时间必须展示，避免实时性变差
        if waited_ms >= self.max_wait_ms:
            return True

        # 规则 2: 单个短句先等 300ms，看 AST 是否马上给出下一片段
        if len(self.pending) == 1:
            source_words = len(first["source"].split())
            if source_words < 5 and waited_ms < self.min_wait_ms:
                return False
            return True

        last = self.pending[-1]
        prev = self.pending[-2]

        # 规则 3: 两句间隔 < 200ms → 视为同一句的碎片，继续合并
        if last["start"] - prev["end"] < 0.2:
            return False

        # 规则 4: 上一句太短 → 更可能是误切，继续等待到 max_wait_ms
        if len(prev["source"].split()) < 5 and waited_ms < self.max_wait_ms:
            return False

        return True

    def _schedule_delayed_commit(self):
        """安排一次延迟提交；若后续片段到达，可复用同一个 pending。"""
        if self.commit_timer and not self.commit_timer.done():
            return
        self.commit_timer = asyncio.create_task(self._delayed_commit())

    async def _delayed_commit(self):
        await asyncio.sleep(self.max_wait_ms / 1000)
        if self.pending:
            self._commit_pending()

    def _commit_pending(self):
        merged = self._merge_pending()
        sentence = self._commit_to_display(merged)
        self.displayed.append(sentence)
        self.pending.clear()

    def _should_trigger_correction(self):
        """纠错触发策略"""
        # 条件 1: 积累 >= 3 句稳定字幕才触发
        if len(self.displayed) < 3:
            return False
        # 条件 2: 距离上次纠错 >= 8 秒，避免字幕频繁抖动
        if time.time() - self.last_correction_time < 8:
            return False
        # 条件 3: VAD 确认说话人暂停
        if self.vad.current_silence_duration() < 0.8:
            return False
        return True

    def _merge_pending(self):
        """合并缓冲区中的多句"""
        source = " ".join(p["source"] for p in self.pending)
        translation = "".join(p["translation"] for p in self.pending)
        return {"source": source, "translation": translation}
```

### 4.5 字幕消息总线（bus/subtitle_bus.py）

**职责**：后端内部模块间解耦。AST 客户端、FlowController、纠错引擎通过 Bus 通信，不直接耦合。

```python
class SubtitleBus:
    """发布-订阅模式的事件总线"""

    def __init__(self):
        self.subscribers: dict[str, list[callable]] = {
            "new_sentence": [],     # AST 输出新句子
            "commit_display": [],   # FlowController 确认展示
            "correction": [],       # 纠错引擎产出修正
            "status_change": [],    # 状态变更
        }

    def publish(self, event_type: str, data: dict):
        for callback in self.subscribers.get(event_type, []):
            asyncio.create_task(callback(data))

    def subscribe(self, event_type: str, callback):
        self.subscribers[event_type].append(callback)
```

### 4.6 纠错引擎（translator/correction.py）

**核心创新**：不是简单"多等几秒再翻译一次"，而是**结构化 Diff + 置信度分级 + 最小粒度修正**。

**技术路线**：使用七牛云 AI 大模型推理平台的 DeepSeek V4 Flash（0.001元/K input, 0.002元/K output），极低成本。

**System Prompt 设计**：

```
你是同声传译的校对专家。你会收到：
1. 最近 N 句英文原文（累积上下文）
2. 之前对每句的快速翻译

请基于更大的上下文，重新审视之前的翻译。
如果某句的快速翻译不够准确，输出修正。

要求：
1. 只修正确实有问题的句子，不要"润色"已经正确的翻译
2. 修正时给出最小改动，不要整句重写
3. 说明修正原因
```

**输出格式**（JSON）：

```json
{
  "corrections": [
    {
      "sentence_id": "s_038",
      "base_version": 1,
      "new_version": 2,
      "old": "我想快速概述一下成就",
      "new": "首先让我快速回顾医疗AI领域的进展",
      "reason": "上下文补全: 'achievements' 在此语境特指医疗AI领域",
      "confidence": 0.92
    }
  ]
}
```

**纠错分级策略**：

| 级别 | 触发条件 | 处理方式 |
|------|----------|----------|
| L1 句内自修正 | AST 同一句 Response 增量更新 | 前端自动覆盖，无动画 |
| L2 跨句语义修正 | 指代消解/术语补全（5~10s延迟） | 删除线 + 高亮动画 |
| L3 段落重构修正 | 段落结束时全段重译（10~15s延迟） | 仅在高置信度时推送，避免大幅改写历史字幕 |

**Revision 约束**：
- 每条字幕都有 `id` 和 `version`，新字幕初始为 `version=1`
- 纠错消息必须携带 `target_id`、`base_version`、`new_version`
- 前端只接受 `base_version` 与当前版本一致的修正，避免旧的异步纠错覆盖新的结果
- `confidence < 0.85` 的修正默认不推送，只记录到开发者日志

### 4.7 SessionState 全局共享状态（controller/session_state.py）

**定位**：一个简单的 `dataclass`，所有模块持有同一份引用。考虑到 `asyncio` 任务会在 `await` 边界交错执行，核心共享字段不随意跨模块直接写入：字幕列表、修正计数、成本统计等写操作统一通过 `FlowController`、`CorrectionEngine` 或 `SessionManager` 的方法完成。

```python
from dataclasses import dataclass, field
from enum import Enum
import time

class SessionPhase(Enum):
    RUNNING = "running"    # 正常翻译中
    PAUSED = "paused"      # 用户暂停 / 静音超时
    STOPPED = "stopped"    # 会话结束

@dataclass
class SessionState:
    """全局共享状态，所有模块持有引用"""
    
    # ── 会话标识 ──
    session_id: str = ""
    phase: SessionPhase = SessionPhase.RUNNING
    started_at: float = 0.0
    
    # ── 翻译统计 ──
    sentence_count: int = 0
    correction_count: int = 0
    total_cost: float = 0.0
    
    # ── AST 连接 ──
    ast_connected: bool = False
    
    # ── VAD 状态 ──
    is_speaking: bool = False
    silence_started_at: float | None = None
    
    # ── 共享数据 ──
    displayed_sentences: list[dict] = field(default_factory=list)  # 每项包含 id/version/source/translation
    glossary: dict[str, str] = field(default_factory=dict)
    
    # ── 便捷属性 ──
    @property
    def silence_duration(self) -> float:
        if self.silence_started_at is None:
            return 0.0
        return time.time() - self.silence_started_at
    
    @property
    def elapsed(self) -> float:
        if self.started_at == 0:
            return 0.0
        return time.time() - self.started_at
```

**使用方式**：在 `main.py` 创建一次，传给所有模块的构造函数。

```python

vad            = VAD(state)
ast_client     = ASTClient(state)
flow_ctrl      = FlowController(state)
correction     = CorrectionEngine(state)
ws_server      = WSServer(state)
# 所有模块内部: self.state = state
```

### 4.8 SessionManager 会话生命周期管理（controller/session_manager.py）

**定位**：管理整个会话的启动/暂停/恢复/停止。**不做复杂状态机**，只有 3 个状态。

```
running ──(用户暂停/静音超时)──→ paused ──(用户恢复/检测说话)──→ running
    │                                │
    └────(用户停止/超时)────→ stopped ←────(用户停止/超时)────┘
```

**核心职责**：

| 场景 | 行为 |
|------|------|
| 用户点击暂停 | 停止发送音频到 AST（但不断开连接），`state.phase = PAUSED` |
| 用户点击恢复 | 恢复发送音频，`state.phase = RUNNING` |
| 静音超过 60 秒 | 自动进入 PAUSED，前端显示"⏸️ 长时间静音，已自动暂停" |
| 暂停后检测到说话 | 自动恢复，进入 RUNNING |
| 用户关闭浏览器 | WebSocket 断连超过 30 秒 → 自动 STOPPED，断开 AST |
| 暂停超过 5 分钟 | 自动 STOPPED，释放 AST 连接 |

```python
class SessionManager:
    def __init__(self, state: SessionState):
        self.state = state
        self._pause_due_to_silence = False
    
    async def pause(self, reason: str = "user"):
        """暂停会话，停止发音频但保持 AST 连接"""
        self.state.phase = SessionPhase.PAUSED
        if reason == "silence":
            self._pause_due_to_silence = True
        # 通知前端
        await bus.publish("status_change", {"state": "paused", "reason": reason})
    
    async def resume(self):
        """恢复会话"""
        self.state.phase = SessionPhase.RUNNING
        self._pause_due_to_silence = False
        await bus.publish("status_change", {"state": "speaking"})
    
    async def stop(self):
        """结束会话，断开 AST，生成统计"""
        self.state.phase = SessionPhase.STOPPED
        # 发送 FinishSession 到 AST
        # 生成最终统计 → 推送前端
        await bus.publish("status_change", {"state": "finished"})
    
    async def _monitor_silence(self):
        """后台协程：持续监控静音时长"""
        while self.state.phase != SessionPhase.STOPPED:
            if self.state.phase == SessionPhase.RUNNING:
                if self.state.silence_duration > 60:
                    await self.pause(reason="silence")
            elif self.state.phase == SessionPhase.PAUSED:
                if self._pause_due_to_silence and self.state.is_speaking:
                    await self.resume()
            await asyncio.sleep(5)  # 每 5 秒检查一次
```

### 4.9 TraceCollector 全链路追踪（utils/trace.py）

**定位**：每个稳定展示的字幕句分配一个 `trace_id`，记录从 AST 句子开始、流控决策到前端渲染的关键时间戳。音频采集阶段只记录最近音频帧时间，不强行把每个 80ms 音频帧映射到最终句子，避免追踪粒度与 AST 断句不一致。

**设计原则**：极简。不做树状嵌套 Span，只有平铺的时间戳数组。

```python
@dataclass
class SentenceTrace:
    """单句全链路追踪"""
    trace_id: str          # 与字幕 id 一致，如 "s_042"
    timestamps: dict[str, float] = field(default_factory=dict)
    
    def mark(self, checkpoint: str):
        """记录一个时间点"""
        self.timestamps[checkpoint] = time.time()
    
    def summary(self) -> str:
        """人类可读的摘要"""
        if len(self.timestamps) < 2:
            return f"{self.trace_id}: incomplete"
        
        keys = list(self.timestamps.keys())
        parts = []
        for i in range(len(keys) - 1):
            a, b = keys[i], keys[i+1]
            delta = (self.timestamps[b] - self.timestamps[a]) * 1000
            parts.append(f"{a}→{b}={delta:.0f}ms")
        return f"{self.trace_id}: " + " | ".join(parts)
    
    @property
    def total_latency_ms(self) -> float:
        """端到端延迟（毫秒）"""
        if "frontend_rendered" in self.timestamps and "audio_captured" in self.timestamps:
            return (self.timestamps["frontend_rendered"] - 
                    self.timestamps["audio_captured"]) * 1000
        return -1


class TraceCollector:
    """收集所有句子的 Trace"""
    
    def __init__(self, max_size: int = 500):
        self.traces: dict[str, SentenceTrace] = {}
        self.max_size = max_size
    
    def start_trace(self, trace_id: str) -> SentenceTrace:
        trace = SentenceTrace(trace_id=trace_id)
        trace.mark("audio_captured")
        self.traces[trace_id] = trace
        # 超过容量，淘汰最旧的
        if len(self.traces) > self.max_size:
            oldest = min(self.traces.keys())
            del self.traces[oldest]
        return trace
    
    def get_trace(self, trace_id: str) -> SentenceTrace | None:
        return self.traces.get(trace_id)
    
    def recent_summaries(self, n: int = 5) -> list[str]:
        """最近 N 句的摘要，供前端展示"""
        recent_ids = list(self.traces.keys())[-n:]
        return [self.traces[tid].summary() for tid in recent_ids]
```

**使用示例**：各模块在关键节点打点。

```python

# controller/flow_controller.py

# server/ws_server.py

# 前端 app.js 收到后

```

### 7.1 正常翻译流（快速通道，延迟 < 1s）

```

                     TranslationSubtitleResponse
                                  ↓
                     TraceCollector.mark("ast_response")
                                  ↓
                     SubtitleBus.publish("new_sentence")
                                  ↓
                     FlowController.on_ast_sentence()
                       (读取 SessionState 做决策)
                                  ↓
                     决策: 立即展示 or 短暂等待 300~800ms 合并碎片
                                  ↓
                     TraceCollector.mark("flow_decision")
                                  ↓
                     WebSocket → 前端 {"type":"subtitle",...}
                                  ↓
                     TraceCollector.mark("ws_pushed")
                                  ↓
                     前端: 新增字幕 → 回传 trace_ack
                                  ↓
```

### 7.2 纠错修正流（慢速通道，延迟 3~15s）

```

  (检查 SessionState.displayed_sentences 数量 + VAD 静音状态)
          ↓
TraceCollector.mark("correction_trigger")
          ↓
CorrectionEngine.run()
          ↓
累积最近 3~8 句稳定字幕的原文 + 快速翻译（从 SessionState 读取）
          ↓
七牛云 AI 推理 DeepSeek V4 Flash
  system: "你是翻译校对专家"
  user:   "原文: {...}  旧翻译: {...}  请检查并修正"
          ↓
解析 response → 校验 sentence_id/version → 结构化 Diff
          ↓
SessionState.correction_count += 1
          ↓
SubtitleBus.publish("correction", corrections)
          ↓
WebSocket → 前端 {"type":"correction",...}
          ↓
前端: 删除线动画 + 高亮新译 + "✓ 已修正" 徽章
```

### 7.3 会话生命周期流

```
用户点击开始 → SessionManager.resume()
  → SessionState.phase = RUNNING
  → 开始发送音频到 AST
  → 前端显示 "🔴 翻译中"

用户点击暂停 → SessionManager.pause(reason="user")
  → SessionState.phase = PAUSED
  → 停止发送音频（不断开 AST）
  → 前端显示 "⏸️ 已暂停"

静音 60 秒 → SessionManager._monitor_silence() 检测到
  → SessionManager.pause(reason="silence")
  → 前端显示 "⏸️ 长时间静音，已自动暂停"

暂停中检测到声音 → SessionManager._monitor_silence() 检测到
  → SessionManager.resume()
  → 前端显示 "🔴 翻译中"

用户关闭浏览器 → ws_server 检测到断连
  → 30 秒内未重连 → SessionManager.stop()
  → 断开 AST，生成最终统计
  → SessionState.phase = STOPPED
```

### 7.4 异常处理流

```
场景 1: AST WebSocket 断连
  → 指数退避重连（1s → 2s → 4s → 8s，最多 5 次）
  → 音频缓冲区保留未发送数据
  → 重连成功后续传
  → SessionState.ast_connected = False → True

场景 2: AST 返回 SessionFailed
  → MVP 阶段: 明确提示 AST 不可用，并停止计费型发送
  → 扩展阶段: 评估 faster-whisper + 七牛云 Chat API 串联翻译
  → 前端显示 "⚠️ 翻译服务不可用 / 已停止"

场景 3: 七牛云 API 超时/限流
  → 纠错通道静默跳过本次纠错
  → 不阻塞快速通道
  → 等下一个触发周期再试

场景 4: 暂停超过 5 分钟
  → SessionManager 自动 STOPPED
  → 释放 AST 连接，省钱
```

## 十二、答辩叙事脚本

> "大家好，我们做的是**AI 同声传译助手**。
>
> 现有同传产品普遍优先保证低延迟，所以字幕通常以前向流式方式展示。它们能很快给出结果，但一旦缺少上下文导致某一句翻译不准，用户很难看到一个清晰、可追踪的"历史字幕修正"过程。
>
> 我们设计了一套**双通道异步架构**解决这个问题：
>
> **快速通道**对接豆包 AST，实现端到端延迟不到 1 秒的实时字幕。
>
> **纠错通道**基于七牛云 AI 推理平台的 DeepSeek V4 Flash，当前后文积累足够后，自动对历史翻译进行回溯校验。我们不是简单重新翻译——而是通过结构化 Diff 引擎定位修改粒度，结合我们设计的 FlowController 流控决策层，只修正确实有问题的句子。
>
> 修正过程在 UI 层有完整的动画呈现：旧翻译删除线渐隐，新翻译高亮滑入，用户始终能感知到系统在做什么。
>
> 我们还有全链路的延迟监控，每一句从 AST 句子开始、流控决策到前端渲染的关键耗时都可追溯。异常情况下会明确提示并停止不必要的计费型发送，保证演示可控。
>
> 这个方案的核心创新在于：**我们不是在模型层面重复造轮子，而是在系统架构层面，把已有 API 的能力重新编排，让实时字幕具备可解释的回溯校对能力。**"

## 十三、风险与应对

| 风险 | 概率 | 应对 |
|------|------|------|
| AST 协议或 Protobuf 编译失败 | 中 | Day 1 优先验证；失败时保留预录音频 Demo，并将 faster-whisper 串联翻译作为扩展方案 |
| 豆包 AST 延迟过大（> 3s） | 中 | 记录 Trace 指标；MVP 不自动切换模型，只提示服务质量下降 |
| 七牛云 API 超时/限流 | 低 | 纠错通道使用令牌桶并静默跳过本轮，不影响快速通道 |
| 回溯纠错误改正确字幕 | 中 | 设置 `confidence >= 0.85` 才推送；前端按 version 校验；Demo 中准备可控测试音频 |
| 音频隐私表述被追问 | 中 | 明确说明：音频不进自建服务端/前端，但会直连第三方 AST 服务 |
| 演示时网络不稳定 | 中 | 准备本地预录 demo 视频作为备用 |
| 3 天时间不够 | 中 | 优先保证核心路径（翻译 + revision 纠错 + 基础指标），TTS/设置面板/完整开发者面板后置 |
| 用户忘记暂停导致 API 持续计费 | 中 | ✅ SessionManager 静音 60s 自动暂停、暂停 5min 自动停止 |
